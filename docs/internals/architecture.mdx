Pear Pro MCP is a small, single-process Node server. Five source files do all the work:

| File | Responsibility |
| --- | --- |
| `src/index.ts` | Entrypoint ‚Äî loads config, builds the MCP server, registers tools, connects stdio |
| `src/config.ts` | Reads & normalizes environment variables into a `PearConfig` |
| `src/client.ts` | Typed HTTP client for the Pear API + auth/refresh |
| `src/tools.ts` | Registers all 21 tools, Zod schemas, and the write guard |
| `src/analytics.ts` | Offline statistical-arbitrage math |
| `src/rebalance.ts` | Rebalance plan fetch + drift analysis |
| `src/monitor.ts` | Standalone background rebalance monitor |

## Request lifecycle

<Steps>
  <Step title="Agent calls a tool">
    The MCP client sends a `tools/call` over stdio with the tool name and arguments.
  </Step>
  <Step title="Schema validation">
    The MCP SDK validates the arguments against the tool's Zod `inputSchema`. Out-of-range or malformed input is rejected here, before any code runs.
  </Step>
  <Step title="Write guard (mutating tools only)">
    `guardWrite(confirm)` checks `PEAR_READ_ONLY`, then `confirm`. A blocked call returns an error and never reaches the network.
  </Step>
  <Step title="HTTP via PearClient">
    Read tools call `client.get(...)` (no auth). Authenticated tools call `client.request(...)`, which ensures a valid JWT first.
  </Step>
  <Step title="Result envelope">
    The result is JSON-stringified into a `text` content block. Errors become `{ text, isError: true }`.
  </Step>
</Steps>

## The `run` / `ok` / `fail` helpers

Every tool body funnels through small helpers in `tools.ts`:

- `ok(data)` ‚Äî wraps a success result as a `text` block (stringifying objects).
- `fail(message)` ‚Äî wraps an error string with `isError: true`.
- `run(fn)` ‚Äî awaits `fn`, returns `ok` on success, and converts thrown errors ‚Äî especially `PearApiError` ‚Äî into a readable `fail`.

This is why every tool returns the same predictable envelope (see [Tool reference ‚Üí overview](/tools/overview)).

## Transport

The server uses `StdioServerTransport`. `stdout` is reserved for the MCP protocol; all human logs (including the readiness line) go to `stderr`:

```
pear-pro-mcp ready | base=<url> | clientId=<id> | mode=READ-ONLY
```

## What runs where

- **The MCP server** (`index.ts`) reacts only to tool calls ‚Äî it never polls or schedules anything.
- **The monitor** (`monitor.ts`) is a *separate* long-running process for scheduled drift checks. It reuses `PearClient` and `rebalance.ts` but does not speak MCP. See [Internals ‚Üí Monitor](/internals/monitor).
