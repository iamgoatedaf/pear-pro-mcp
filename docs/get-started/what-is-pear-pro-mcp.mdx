## The one-paragraph version

Pear Pro MCP is a thin, typed MCP server that exposes the [Pear Trading API](https://docs.pearprotocol.io) to AI agents. Pear already ships an in-house quant ("Agent Pair"). This server exposes the same capability **outbound**: any agent can read pair/basket stats, screen trade ideas, reason over z-scores, and — when you opt in — execute and manage consolidated positions. The goal is to make Pear the default execution venue in the agentic-trading meta.

## What is pair / basket trading?

A **pair trade** goes long one asset and short another (e.g. long ETH / short BTC) so you are exposed to the *ratio* between them rather than the market's absolute direction. A **basket** generalizes this to many weighted legs on each side. Pear executes and manages the whole thing as a single **consolidated position** with one entry, one set of TP/SL rules, and one rebalance flow.

Pear Pro MCP speaks this model natively:

- `longAssets` / `shortAssets` accept multiple **weighted** legs.
- TP/SL supports `WEIGHTED_RATIO` thresholds that make sense for baskets, not just single prices.
- Rebalancing keeps each leg at its target weight as prices drift.

## What the server adds on top of the raw API

<Steps>
  <Step title="A safety-gated tool catalog">
    Fund-moving tools are guarded by a global read-only switch **and** a per-call `confirm: true`. By default nothing can move money.
  </Step>
  <Step title="Validated, bounded inputs">
    Every tool input is a [Zod](https://zod.dev) schema: leverage is clamped to 1–100, slippage to 0.1%–10%, weights to 0.0001–1.0, and execution/trigger types to known enums. Bad input is rejected before any HTTP call.
  </Step>
  <Step title="Non-mutating rebalance preview + drift math">
    `pear_rebalance_plan` and `pear_rebalance_check` compute what *would* trade and how far each leg has drifted — safe to poll on a schedule without ever placing an order.
  </Step>
  <Step title="Offline statistical-arbitrage helpers">
    `pear_pair_analytics` computes correlation, beta, spread z-score, rolling z-score and annualized vol with no API round-trip.
  </Step>
  <Step title="A background rebalance monitor">
    A separate process polls your positions, computes drift, and either alerts you (default) or auto-rebalances (opt-in). See the [monitor guide](/guides/rebalance-monitor).
  </Step>
</Steps>

```
                                          │

```

The agent never holds your API key — it only sends tool calls over stdio. The server is the only thing that talks to Pear and signs requests. See [Internals → Architecture](/internals/architecture) for the full request lifecycle.

## Which engine does it drive?

The documented endpoints target the **Hyperliquid v2** engine (`https://hl-v2.pearprotocol.io`). The **Pear Pro x Lighter** engine should expose the same shapes; if the base URL or auth differs, only the HTTP client and `PEAR_API_BASE_URL` need changing. A partner `clientId` (instead of `APITRADER`) lets Pear attribute volume/fees from agents routed through this MCP.
